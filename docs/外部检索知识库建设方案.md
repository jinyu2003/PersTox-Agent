# PersTox-Agent 外部检索知识库建设方案

## 1. 建设目标

PersTox-Agent 的外部检索知识库不是简单汇总多个数据库，而是要服务于两个核心任务：

1. 通用毒性预测  
面向药物本身，回答该药是否具有器官毒性倾向，以及毒性与哪些结构、理化性质、靶点、通路有关。

2. 个性化毒性预测  
面向药物与患者画像，回答在特定患者中风险为何上升或下降，以及风险变化由哪些患者因素驱动。

因此，知识库应围绕统一机制链组织：

`药物 -> 代谢/代谢物 -> 靶点/脱靶 -> 通路扰动 -> ADE/器官毒性 -> 患者修饰因素`

---

## 2. 总体设计原则

### 2.1 三层定位

建议将外部检索知识库拆为三个子系统：

1. 真值裁决库  
用于事实核查、评测、冲突裁决，定义“什么算真”。

2. 机制检索库  
用于解释“药物为什么有毒”，支持药物侧结构、性质、靶点、通路归因。

3. 个性化规则库  
用于解释“患者为什么更危险”，支持 PGx、共病、合并用药、器官功能修正。

### 2.2 两阶段服务对象

知识库设计必须直接对应 Agent 的两阶段输出：

- 第一阶段：通用毒性预测  
核心是药物侧基线风险与机制解释。

- 第二阶段：个性化毒性预测  
核心是在第一阶段基线上叠加患者修正因素，输出最终个体风险。

### 2.3 建设顺序

建议先做肝毒与心毒两个 MVP 器官，再扩展到 8 个 SOC，原因是：

- 数据最充分
- 金标准最清晰
- 机制链更完整
- 更适合先跑通评测闭环

---

## 3. 金标准层设计

## 3.1 权威来源分级

建议按以下等级定义事实裁决依据：

| 层级 | 类型 | 用途 |
|---|---|---|
| L1 | 监管/标签级：FDA 说明书、EMA SmPC、NMPA 说明书 | 最高权威，可直接作为 ground truth |
| L2 | 指南/共识：CPIC、DPWG、ESC cardio-oncology、ASCO/ESMO、DILI 指南 | 个性化规则与临床判断核心来源 |
| L3 | 权威 curated 库/系统综述：LiverTox、CredibleMeds、DILIrank、Cochrane | 专科高可信证据 |
| L4 | 教科书、原始文献、专家经验 | 用于机制补充，不宜单独裁决 |
| L5 | FAERS、自发报告、个案报告 | 仅作信号，不能单独判真 |

### 3.2 裁决门槛

- 一条毒性结论要判定为“成立”，至少需要 1 条 L1-L3 证据支持。
- 若仅有 L5 信号支持，则标记为 `unverified`，不能视作真值。

### 3.3 金标准判定维度

每条结论建议按以下维度独立裁决：

| 维度 | 判定要求 |
|---|---|
| 关联存在性 | 是否确有该毒性 |
| 器官归属 | 是否归到正确 SOC |
| 严重度 | CTCAE 分级是否合理 |
| 频率量级 | 常见/少见/罕见是否大致正确 |
| 机制归因 | 机制解释是否合理 |
| 条件性 | 是否遗漏剂量、人群、基因型等关键条件 |

建议采用四档标签：

- `correct`
- `partially_correct`
- `incorrect`
- `unverified`

### 3.4 金标准数据集建设建议

MVP 阶段先建立肝毒和心毒的 adjudicated gold cases，建议 30-50 条起步，覆盖：

- 正控药物
- 负控药物
- PGx 条件阳性案例
- 条件性毒性案例
- 每条案例附 L1-L3 来源与日期

负控建议：

- 肝毒负控：LiverTox 中明确 unlikely cause 或 category E 的药物
- 心毒负控：不在 CredibleMeds 风险清单中的药物

## 4. 检索数据源分层与可信度

## 4.1 可信度分层

建议统一采用四级可信度体系：

| 等级 | 含义 | 用法 |
|---|---|---|
| Tier 1 | 金标准/指南级 | 可作 ground truth 或强规则依据 |
| Tier 2 | 高质量人工策展 | 可作高可信支持证据 |
| Tier 3 | 聚合/计算预测 | 可作弱支持或机制补充 |
| Tier 4 | 自发报告/偏倚较强 | 仅作信号，不单独判真 |

## 5. 外部检索知识库模块设计

## 5.1 药物基础信息模块

作用：统一药物身份、别名、分类、结构、代谢、DDI、适应症。

| 数据源 | 可信度 | 主要用途 |
|---|---|---|
| RxNorm | T1 | 药物标准命名 |
| ATC | T1 | 药物分类 |
| DrugBank | T2 | 药物、代谢、DDI、靶点、适应症 |
| PubChem | T2.5 | 化学结构与聚合信息 |
| ChEMBL | T2.6 | 开放 bioactivity、ADME assay |
| admetSAR 3.0 | 实验值 T2，预测值 T3 | ADMET 标签与性质端点 |

建议核心字段：

- `drug_id`
- `drugbank_id`
- `rxnorm_id`
- `atc_code`
- `drug_name`
- `aliases`
- `smiles`
- `inchi_key`
- `drug_type`
- `metabolism_summary`
- `ddi_summary`

## 5.2 蛋白/基因模块

作用：支撑“药物 -> 靶点 -> 通路 -> 毒性机制”以及“患者基因型 -> 风险修正”。

| 数据源 | 可信度 | 主要用途 |
|---|---|---|
| UniProt Swiss-Prot | T1 | 蛋白标准实体 |
| HGNC | T1 | 基因命名标准 |
| NCBI Gene | T1 | 基因基础信息 |
| Ensembl | T2 | 基因注释 |
| PDB | T2 | 实验结构 |
| AlphaFold DB | T3 | 预测结构 |

建议核心字段：

- `gene_symbol`
- `hgnc_id`
- `ncbi_gene_id`
- `uniprot_id`
- `protein_name`
- `function_summary`
- `organ_expression`
- `pgx_relevance`

## 5.3 通路模块

作用：为毒性机制解释提供“下游扰动”层。

| 数据源 | 可信度 | 主要用途 |
|---|---|---|
| Reactome | T2 | 开放 curated 通路 |
| KEGG | T2 | 通路与药物映射 |
| Gene Ontology | T2 | 生物过程/分子功能 |
| WikiPathways | T2-T3 | 社区补充 |
| MSigDB | T2-T3 | 富集分析基因集 |

建议核心字段：

- `pathway_id`
- `pathway_source`
- `pathway_name`
- `member_genes`
- `toxicity_relevance`
- `evidence_level`

## 5.4 不良反应与术语模块

作用：统一 ADE 命名、映射到 SOC、支持 CTCAE 对齐。

| 数据源 | 可信度 | 主要用途 |
|---|---|---|
| MedDRA | T1 | ADE 标准术语与 SOC 映射 |
| CTCAE v5.0 | T1 | 严重度分级标准 |
| ADReCS | T2 | ADR 本体与机制链 |
| OnSIDES | T2-T3 | 标签抽取 |
| SIDER | T3 | 已冻结，尽量不作核心源 |

建议核心字段：

- `ade_id`
- `meddra_pt`
- `meddra_soc`
- `ctcae_term`
- `ctcae_grade_mapping`
- `mechanism_annotation`

## 5.5 器官毒性专病模块

这是知识库中最关键的一层，建议独立于一般 ADE 表建设。

### 5.5.1 肝毒模块

| 数据源 | 可信度 | 作用 |
|---|---|---|
| DILIrank | T1 | 肝毒金标准标签 |
| LiverTox | T1 | 临床专论与 likelihood |
| DILIst | T1-T2 | 结构化 DILI 数据 |
| LTKB | T1-T2 | FDA 肝毒知识库 |
| Open TG-GATEs | T2 | 表达与机制证据 |
| ToxCast/Tox21 | T2 | 体外机制支持 |

### 5.5.2 心毒模块

| 数据源 | 可信度 | 作用 |
|---|---|---|
| CredibleMeds | T1 | QT/TdP 风险金标准 |
| ChEMBL hERG 数据集 | T2 | hERG 阻滞实验依据 |
| ToxCast/Tox21 | T2 | 心毒相关机制支持 |
| 说明书/标签信息 | T1 | 心衰、心肌病等补充依据 |

### 5.5.3 其他器官模块

| 器官 | 主要数据源 | 建议阶段 |
|---|---|---|
| 肾 | TG-GATEs、ToxCast、CTD | Phase 2 |
| 血液 | ToxCast、CTD | Phase 2 |
| 神经 | ToxCast、CTD | Phase 2 |
| 皮肤 | LLNA/OECD、标签、病例 | Phase 2 |
| 免疫/irAE | ICI 试验 CTCAE SAE | Phase 2 |
| 消化道 | FAERS | 仅推理调用，不入训练 |

## 5.6 药物-靶点-通路机制模块

作用：把“毒性标签”转成可解释机制链。

| 关系类型 | 数据源 | 可信度 |
|---|---|---|
| 药物-靶点 | IUPHAR/Guide to Pharmacology | T1-T2 |
| 药物-靶点 | ChEMBL | T2 |
| 药物-靶点 | BindingDB | T2-T3 |
| 药物-靶点 | DrugBank | T2 |
| 药物-靶点 | STITCH | T3 |
| 药物-通路 | PharmGKB pathways | T1-T2 |
| 药物-通路 | KEGG/Reactome/SMPDB | T2 |
| 蛋白-通路 | Reactome/KEGG/GO | T2 |
| 蛋白-通路 | STRING | T3 |

注意：

- `药物 -> 通路` 很多情况下是通过 `药物 -> 靶点 -> 通路` 推导得到
- 这类衍生关系必须保留推导链和来源，不能和原始事实混用

## 5.7 个性化修饰模块

作用：支撑第二阶段个性化毒性预测。

| 数据源 | 可信度 | 主要用途 |
|---|---|---|
| CPIC | T1 | 基因型到临床建议 |
| DPWG | T1 | PGx 规则补充 |
| PharmGKB | T1-T2 | PGx 证据与通路 |
| ClinVar | T1-T2 | 变异临床意义 |
| Open Targets | T2 | 基因-疾病证据 |
| CTD | T2-T3 | 化学-基因-疾病推断 |
| DDI 规则库 | T1-T2 | 合并用药修正 |
| Beers/器官功能规则 | T1-T2 | 老年与功能状态修正 |

建议统一为四类个性化因素：

- `PGx`
- `comorbidity`
- `comedication`
- `organ_function`

## 6. 标准化实体与主键统一

建议统一主键如下：

| 实体 | 主键标准 |
|---|---|
| 药物 | DrugBank ID 为主，辅以 RxNorm、ATC、PubChem CID |
| 基因 | HGNC symbol / HGNC ID |
| 蛋白 | UniProt ID |
| 通路 | Reactome ID / KEGG ID |
| ADE | MedDRA PT |
| 器官系统 | MedDRA SOC |
| 疾病/共病 | ICD-10/11 |
| 变异/基因型 | PharmGKB allele / ClinVar / 标准星号命名 |

统一原则：

- 所有外部数据入库前必须先做 ID mapping
- 原始来源 ID 不删除，但只作为 `xref`
- 检索和推理统一基于标准主键

## 7. 证据对象与可信度表达

建议知识库中的每条结论都挂统一证据对象，而不是只保留原始文本。

示例：

```json
{
  "evidence_id": "EV_0001",
  "claim_type": "drug_causes_hepatotoxicity",
  "subject": "drugbank:DB00316",
  "object": "meddra:Hepatobiliary disorders",
  "source": "LiverTox",
  "tier": "T1",
  "level": "L3",
  "support_type": "direct",
  "date": "2025-01-10",
  "pmid": "xxxxxx",
  "confidence_score": 0.92,
  "verdict_role": "ground_truth|supporting|signal_only"
}
```

推荐规则：

- Tier 1 + L1/L2：可直接作 ground truth
- Tier 2 + L3：可作高可信 supporting evidence
- Tier 3：用于机制补充、弱支持
- Tier 4 + L5：仅作 signal，不单独判真

## 8. 面向两阶段任务的检索组织方式

## 8.1 第一阶段：通用毒性预测

目标：回答“药物本身有何器官毒性风险，为什么”。

建议检索顺序：

1. 药物标准化  
将药名、SMILES、DrugBank ID 统一到内部药物主键。

2. 检索通用毒性标签  
从 DILIrank、LiverTox、CredibleMeds、PersADE、说明书等来源获取已知毒性谱。

3. 检索结构与 ADMET 特征  
从 admetSAR 3.0、ChEMBL ADME、结构警示库中获取性质特征。

4. 检索药物-靶点关系  
从 DrugBank、ChEMBL、IUPHAR、BindingDB 获取靶点与活性。

5. 检索靶点-通路-器官机制  
从 Reactome、KEGG、GO、ADReCS 获取机制链。

6. 组装药物侧解释链  
形成 `结构/性质 -> 靶点/脱靶 -> 通路 -> 器官毒性` 的解释路径。

## 8.2 第二阶段：个性化毒性预测

目标：回答“为什么该患者风险更高或更低”。

建议检索顺序：

1. 读取第一阶段基线风险  
将通用毒性结果作为 baseline risk。

2. 检索 PGx 规则  
从 CPIC、DPWG、PharmGKB、ClinVar 获取基因型修正信息。

3. 检索共病修正规则  
识别基础疾病是否增加易感性或改变风险解释。

4. 检索合并用药风险  
重点处理 CYP 抑制/诱导、QT 叠加、转运体竞争、已知 DDI。

5. 检索器官功能修正规则  
如 eGFR、ALT/AST、LVEF 等指标导致的风险变化。

6. 组装患者侧解释链  
形成 `患者因素 -> 暴露/药效/免疫易感性变化 -> 风险上调/下调` 的解释路径。

## 9. 推荐的数据表与图谱结构

## 9.1 实体表

建议至少包含以下实体表：

- `drug`
- `gene`
- `protein`
- `pathway`
- `ade`
- `organ_soc`
- `disease`
- `variant_or_allele`
- `evidence_source`
- `clinical_rule`

## 9.2 关系表

建议至少包含以下关系表：

- `drug_target`
- `drug_metabolism`
- `drug_transporter`
- `drug_pathway`
- `protein_pathway`
- `drug_ade`
- `ade_soc`
- `gene_drug_pgx`
- `drug_ddi`
- `gene_disease`
- `drug_disease`
- `rule_patient_factor_risk_shift`

## 9.3 证据表

建议单独建设 `evidence` 表，并让所有关系通过 evidence 挂接：

- `evidence_id`
- `relation_type`
- `source_name`
- `tier`
- `date`
- `directness`
- `pmid_or_url`
- `confidence_score`
- `quote_or_summary`
- `applicable_condition`

这样做的作用：

- 便于溯源
- 便于冲突裁决
- 便于知识审计
- 便于评测与回放证据链

## 11. MVP 建设顺序建议

## Phase 1：先做肝毒与心毒

建议优先纳入以下数据源：

- 肝毒：DILIrank、LiverTox、DILIst、LTKB、Open TG-GATEs
- 心毒：CredibleMeds、ChEMBL hERG、说明书标签
- 药物基础：DrugBank、RxNorm、ATC、admetSAR 3.0
- 机制层：ChEMBL、IUPHAR、Reactome、GO、ADReCS
- 个性化层：CPIC、DPWG、PharmGKB、DDI 规则

Phase 1 目标：

- 跑通两阶段检索闭环
- 建立金标准裁决体系
- 建立肝毒/心毒 benchmark
- 支撑第一版药物知识卡片与证据链输出

## Phase 2：扩展到 8 个 SOC

建议扩展顺序：

1. 肾
2. 血液
3. 神经
4. 皮肤
5. 免疫/irAE
6. 消化道

## 12. 最终建议

从工程和临床结合角度看，PersTox-Agent 的外部检索知识库最合理的形态不是“一个大而全的库”，而是：

- 用真值裁决库保证事实核查与评测闭环
- 用机制检索库支撑药物侧毒性解释
- 用个性化规则库支撑患者侧风险修正

整体上，建议先聚焦肝毒和心毒，围绕以下主链建设：

`药物 -> 机制 -> 器官毒性 -> 患者修正`

并要求每条结论都带有：

- 来源等级
- 证据强度
- 适用条件
- 日期
- 可追溯出处

一句话概括：

**PersTox-Agent 外部检索知识库应以“金标准裁决 + 机制检索 + 个性化规则”三层架构为核心，先以肝毒和心毒跑通两阶段预测闭环，再逐步扩展到全器官系统。**
